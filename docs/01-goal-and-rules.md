# Цель и правила

Зачем нужны 100 публичных коммитов и как добиваться их честно.

## Цель

- **100 публичных коммитов** на default-ветке (обычно `main`) в публичном репозитории на GitHub.
- Цель — показать стабильную практику работы с Git и историей проекта, а не «накрутить счётчик».

## Правила (без накрутки)

- **Нельзя:** пустые коммиты (`--allow-empty`), подделка дат (backdate), спам-правки (типа «fix typo» по 50 файлам), автогенерация мусора, бессмысленные «fix/update».
- **Можно:** атомарные коммиты (одна мысль — один коммит), реальные изменения: документация, структура, скрипты, правки по смыслу.
- Каждый коммит = заметная полезная порция работы (документация, шаблоны, политики, CI, скрипты, структура).

## Как работать безопасно

1. Делай коммиты по плану (например, по гайдам в `docs/` или по задачам в PROGRESS.md).
2. Перед коммитом: `git status`, `git diff` — убедись, что в коммит попадает только нужное.
3. После коммита: `git log -1 --stat` — проверь, что коммит содержит ожидаемые файлы.
4. Регулярно делай `git push`, чтобы история была на GitHub и коммиты считались публичными.

## Итог

100 коммитов — это про привычку и качество истории, а не про количество любой ценой.
