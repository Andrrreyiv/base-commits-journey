# Issues и метки (labels): workflow

Как вести задачи через Issues и использовать метки для навигации.

## Цель

- Создавать issue под задачу и связывать с PR.
- Использовать метки (bug, enhancement, docs и т.д.) для фильтрации и приоритета.

## Создание issue

- В репозитории: вкладка **Issues** → **New issue**. Выбери шаблон (bug report / feature request), если настроены.
- Заполни заголовок и описание: что сделать, зачем, как проверить.
- Назначь метки (Labels): например `docs`, `enhancement`, `good first issue`.

## Связь issue и PR

- В описании PR укажи: `Closes #5` или `Fixes #5` — после merge PR issue #5 автоматически закроется.
- В коммитах можно писать «Fix #5» — GitHub свяжет коммит с issue.

## Рекомендуемые метки

- **bug** — ошибка в коде или документации.
- **enhancement** / **feature** — новая возможность.
- **docs** — только документация.
- **chore** — настройки, конфиги, без изменения логики.
- **good first issue** — подходит для новичков.
- **priority: high** / **priority: low** — при необходимости.

Настройка: **Settings** → **Labels** (или см. гайд [Рекомендуемые метки](25-labels-reference.md)).

## Workflow

1. Создай issue с описанием задачи и метками.
2. Создай ветку (например `fix/issue-5`), внеси изменения.
3. Открой PR, в описании укажи `Closes #5`.
4. После ревью и merge PR issue закроется автоматически.

## Проверка

- В issue виден связанный PR; после merge issue в статусе Closed.
- По меткам можно фильтровать: **Labels** в списке Issues.

## Типовые ошибки

- **Не указал Closes #N** — issue останется открытым после merge PR; добавь в описание PR.
- **Слишком общий заголовок** — «Исправить» не помогает при поиске; лучше «Исправить ссылку в docs/README».
